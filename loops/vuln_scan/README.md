# vuln_scan — exploit-verified vulnerability scanner

**Who it's for / what it saves you:** a security engineer or pentester gets back only findings that came with a live working exploit attached — zero time spent triaging false alarms.

THE FLAGSHIP loop. A static scanner produces a list of *guesses*. This loop
upgrades every guess to a *fact* (or throws it away) by firing a real exploit at
a live target. **The model proposes; the canary decides.**

## The verifier (the product)

For each suspected sink, `verifier.py` does exactly this:

1. **Mints a fresh canary** — `"CANARY_" + secrets.token_hex(16)`. Unguessable,
   unique per finding, generated by the verifier *after* the scanner has spoken.
   The scanner never sees it.
2. **A cheap model proposes ONE HTTP request** (`claude-haiku-4-5`) whose payload
   embeds that canary such that, *if the sink is genuinely exploitable*, the
   server is forced to emit the canary in its response. (For command injection:
   `host=127.0.0.1; echo CANARY`. For `eval`: `expr='CANARY'`.) The model writes
   only the request — it renders no verdict.
3. **The verifier FIRES that request** at the live demo app on `127.0.0.1` with
   the `requests` library and reads the response body.
4. **Confirmed iff the exact canary substring appears in the body.** Full stop.

```
Finding (guess) --strong model scan--> proposer (cheap model) writes request
                                                 |
                                        verifier FIRES it at live app
                                                 |
                          canary in response body?  yes -> CONFIRMED, no -> rejected
```

The proposing model and the scanning model are different calls in different
contexts, and **neither is ever asked "is this real?"** — the running process
answers that. The proof (exact request, canary, HTTP status, response snippet)
lands in `Verdict.evidence` so a human can re-fire it and confirm in under a
minute.

## Why it pays

- **Direct revenue / trust:** a scan you can hand a customer says "here is the
  request, here is the canary in the response" — not "our model thinks line 71
  looks risky." Exploit-verified findings have a ~0% false-positive rate by
  construction, which is the entire value of a paid pentest.
- **Tirelessness:** it re-scans and re-verifies every round until it goes dry,
  with a hard USD budget ceiling. It never gets bored on finding #40.
- **Right model at each tier:** the open-ended source reasoning gets the strong
  model; the mechanical "write one request" step gets the cheap one. Cost scales
  with the cheap tier, not the expensive one.
- **Parallelism-ready:** each finding is independent; the dedupe key
  (`file:line:param`) means a rejected finding is never re-tested.

## The gaming trap

Confirmation is the canary in the response body, **full stop**. The loop closes
the obvious cheats in code, not just prose:

- A model that outputs `"exploited!"` or a confident rationale — **ignored.**
  Only the canary substring in the live response counts.
- A 500 error or a stack trace with no canary — **FAILS** (it never ran).
- An endpoint that blindly mirrors any input would let a reflected canary fake a
  match. The verifier closes this by inspecting the **very request it fired**: if
  any attacker-controlled payload value (canary *plus* its injection scaffolding —
  the `; echo`, the `UNION SELECT`, the quotes) comes back **verbatim** in the
  response, the endpoint merely echoed input, so the canary proves mirroring, not
  execution — **rejected** (`_is_pure_reflection`). Under genuine execution the
  scaffolding is consumed and only the canary surfaces, so a real injection still
  confirms. Unlike a separate benign probe, this cannot miss the actual
  vulnerable param.
- The proposed payload must actually contain the canary
  (`payload_carried_canary`), or the proposer is guessing and any match is
  coincidence — **rejected.**

All three gate flags are recorded in `Verdict.evidence["checks"]`
(`canary_in_body`, `payload_carried_canary`,
`payload_reflected_verbatim_no_execution`).

## Runnable status

**Runnable — out-of-the-box.** Needs only `ANTHROPIC_API_KEY`. The bundled
`demo_app.py` (a deliberately vulnerable `http.server` + `sqlite3` app, standard
library only) is started as a subprocess on `127.0.0.1` before the scan and torn
down after.

```bash
export ANTHROPIC_API_KEY=sk-ant-...
python -m loops.vuln_scan
```

Optional env knobs: `VULN_SCAN_MAX_USD` (default `2.0`),
`VULN_SCAN_MAX_ROUNDS` (default `8`), `VULN_SCAN_RUN_DIR` (default
`./.vuln_scan_run`). Confirmed exploits and their evidence are appended to
`<run_dir>/confirmed.jsonl`.

> ⚠️ `demo_app.py` is intentionally vulnerable (command injection, SQLi, eval,
> reflected output). It binds to `127.0.0.1` only. Do not deploy it.

## Files

- `verifier.py` — the canary-decides verifier (read this first).
- `demo_app.py` — the deliberately vulnerable target.
- `loop.py` — `build(...)` wiring + `main()`, plus the subprocess lifecycle.
- `__main__.py` — `python -m loops.vuln_scan`.
